## 1. 系统启动问题

### 1.1 程序无法启动

**症状**: 双击程序图标无响应，或闪退

**可能原因**:
1. Qt运行库缺失或版本不匹配
2. 配置文件损坏
3. 权限不足
4. DLL依赖缺失

**诊断步骤**:
```batch
# 1. 检查日志文件
type logs\app_2026-04-23.log

# 2. 检查DLL依赖
dumpbin /dependents appFlipGearboxFactoryTest.exe

# 3. 以管理员身份运行
右键 -> 以管理员身份运行
```

**解决方法**:
- **方法1**: 重新安装Qt 6.11.0运行库
- **方法2**: 恢复默认配置文件
- **方法3**: 检查并安装缺失的DLL
- **方法4**: 以管理员身份运行程序

---

### 1.2 配置文件加载失败

**症状**: 启动时提示"CONFIGURATION LOAD FAILURE"

**日志示例**:
```
[2026-04-23 10:30:15.123] ERROR - Failed to load station config from: config/station.json
[2026-04-23 10:30:15.124] ERROR - JSON parse error: illegal value
```

**诊断步骤**:
1. 检查配置文件是否存在
2. 验证JSON格式是否正确
3. 检查文件编码（应为UTF-8）

**解决方法**:
```batch
# 1. 验证JSON格式
python -m json.tool config/station.json

# 2. 恢复默认配置
copy config\station.json.backup config\station.json

# 3. 手动修正错误
使用文本编辑器打开config/station.json，修正语法错误
```

---

### 1.3 字体加载失败

**症状**: 界面显示方块字符或默认字体

**日志示例**:
```
[2026-04-23 10:30:16.456] WARN - Failed to load font: fonts/HarmonyOS_Sans_SC_Regular.ttf
```

**解决方法**:
1. 确认`fonts/`目录存在
2. 复制字体文件到正确位置
3. 检查字体文件权限

## 2. 设备连接问题

### 2.1 所有设备连接失败

**症状**: 设备诊断页面所有设备显示红色

**可能原因**:
1. 串口驱动未安装
2. 串口被其他程序占用
3. 硬件未供电

**诊断步骤**:
```batch
# 1. 查看可用串口
mode

# 2. 检查设备管理器
devmgmt.msc
```

**解决方法**:
- 安装USB转串口驱动（CH340/FTDI）
- 关闭占用串口的程序
- 检查设备电源

---

### 2.2 单个设备连接失败

**症状**: 某个设备状态显示红色，其他正常

**日志示例**:
```
[2026-04-23 10:35:20.789] ERROR [ModbusRtuBusController.cpp:145] - Timeout waiting for response from slave 2
```

**诊断步骤**:
1. 检查该设备的串口配置
2. 使用串口调试工具测试
3. 检查Modbus地址设置

**串口测试命令**:
```batch
# 配置串口参数
mode COM4: BAUD=19200 PARITY=N DATA=8 STOP=2

# 使用Modbus Poll工具测试
# 读取保持寄存器: 功能码03, 地址0000, 数量0002
```

**解决方法**:
- 修正配置文件中的串口号
- 检查波特率、校验位、停止位设置
- 确认设备Modbus地址
- 检查串口线连接

---

### 2.3 间歇性连接失败

**症状**: 设备时而连接成功，时而失败

**日志示例**:
```
[2026-04-23 10:40:15.123] WARN - CRC error in response from slave 3
[2026-04-23 10:40:18.456] INFO - Successfully read from slave 3
[2026-04-23 10:40:21.789] WARN - CRC error in response from slave 3
```

**可能原因**:
1. 串口线质量差或接触不良
2. 电磁干扰
3. 波特率设置过高
4. 设备故障

**解决方法**:
- 更换高质量屏蔽串口线
- 远离干扰源（变频器、大功率电机）
- 降低波特率（如从115200降至9600）
- 增加超时时间
- 添加铁氧体磁环

---

## 3. 测试执行问题

### 3.1 测试无法启动

**症状**: 点击"开始测试"按钮无响应

**可能原因**:
1. 配方未选择
2. 序列号未输入
3. 设备未连接
4. 上次测试未完成

**诊断步骤**:
1. 检查配方选择框
2. 检查序列号输入框
3. 查看设备状态
4. 查看日志文件

**解决方法**:
- 选择有效配方
- 输入序列号
- 确保所有设备连接正常
- 重启应用程序

---

### 3.2 归零阶段失败

**症状**: 测试在归零阶段失败

**日志示例**:
```
[2026-04-23 11:00:15.123] ERROR - Homing failed: Magnet not detected within timeout
```

**可能原因**:
1. 编码器零点磁铁未安装
2. 电机转速过慢
3. 超时时间设置过短
4. 编码器故障

**解决方法**:
- 检查编码器零点磁铁位置
- 增加归零速度（修改配方）
- 延长超时时间（修改配方）
- 手动旋转齿轮箱，观察编码器读数

---

### 3.3 角度阶段失败

**症状**: 测试在角度阶段失败，提示角度偏差过大

**日志示例**:
```
[2026-04-23 11:05:30.456] ERROR - Angle phase failed: Target=360.0°, Actual=355.2°, Error=4.8°
```

**可能原因**:
1. 编码器分辨率不足
2. 机械间隙过大
3. 速度控制不稳定
4. 配方参数设置过严

**解决方法**:
- 检查编码器分辨率设置
- 检查机械连接
- 降低目标速度
- 放宽角度容差（修改配方）

---

### 3.4 负载阶段失败

**症状**: 测试在负载阶段失败，提示扭矩超出范围

**日志示例**:
```
[2026-04-23 11:10:45.789] ERROR - Load phase failed: Torque=35.2Nm exceeds max=30.0Nm
```

**可能原因**:
1. 制动电压过高
2. 齿轮箱质量问题
3. 配方参数设置不当
4. 传感器校准偏差

**解决方法**:
- 降低制动电压（修改配方）
- 检查齿轮箱质量
- 调整扭矩范围（修改配方）
- 校准扭矩传感器

---

### 3.5 电流过载

**症状**: 测试失败，提示电流超限

**日志示例**:
```
[2026-04-23 11:15:20.123] ERROR - Motor current=6.5A exceeds limit=5.0A
```

**可能原因**:
1. 负载过大
2. 电机堵转
3. 电流限制设置过低
4. 电机故障

**解决方法**:
- 降低制动电压
- 检查机械卡滞
- 提高电流限制（修改配方）
- 检查电机状态

---

## 4. 数据采集问题

### 4.1 实时曲线不更新

**症状**: 测试执行页面的实时曲线静止不动

**可能原因**:
1. 采集调度器未启动
2. 设备通信失败
3. 界面刷新问题

**诊断步骤**:
1. 查看日志中的采集错误
2. 检查设备连接状态
3. 重启应用程序

**解决方法**:
- 检查采集调度器状态
- 确保设备通信正常
- 重启应用程序

---

### 4.2 数据跳变异常

**症状**: 实时曲线出现尖峰或突变

**日志示例**:
```
[2026-04-23 11:20:15.123] WARN - Torque reading jumped from 10.2Nm to 150.5Nm
```

**可能原因**:
1. 传感器故障
2. 通信干扰
3. CRC校验失败但未检测到

**解决方法**:
- 检查传感器连接
- 改善电磁屏蔽
- 添加数据滤波（需修改源码）

---

### 4.3 缓冲区溢出

**症状**: 长时间测试后数据丢失

**日志示例**:
```
[2026-04-23 11:25:30.456] WARN - Telemetry buffer full, oldest data will be discarded
```

**解决方法**:
- 缩短测试时间
- 增加缓冲区大小（需修改源码）
- 降低采集频率

---

## 5. 报告生成问题

**症状**: 生成的JSON报告缺少部分数据

## 6. 性能问题

### 6.1 CPU占用过高

**症状**: 系统运行缓慢，CPU占用率持续>80%

**可能原因**:
1. 采集频率过高
2. 日志输出过多

**解决方法**:
- 降低采集频率（修改配置）
- 减少日志输出（设置日志级别）
- 降低界面刷新率

---

### 6.2 内存占用持续增长

**症状**: 长时间运行后内存占用不断增加

**可能原因**:
1. 内存泄漏
2. 缓冲区未清理
3. 日志文件过大

**诊断步骤**:
1. 运行稳定性测试
2. 监控内存使用
3. 查看日志文件大小

**解决方法**:
- 定期重启应用程序
- 清理旧日志文件
- 报告内存泄漏问题

## 7. 日志分析

### 7.1 日志级别说明

| 级别 | 含义 | 示例 |
|------|------|------|
| DEBUG | 调试信息 | 详细的函数调用、变量值 |
| INFO | 一般信息 | 系统启动、配置加载 |
| WARN | 警告信息 | 通信超时、数据异常 |
| ERROR | 错误信息 | 设备连接失败、测试失败 |
| FATAL | 致命错误 | 系统崩溃 |

### 7.2 常见日志模式

#### 设备连接超时
```
[2026-04-23 10:00:15.123] ERROR [ModbusRtuBusController.cpp:145] - Timeout waiting for response from slave 2
```
**含义**: Modbus从站2无响应  
**检查**: 串口连接、设备电源、Modbus地址

#### CRC校验失败
```
[2026-04-23 10:00:20.456] WARN [ModbusFrame.cpp:89] - CRC mismatch: expected=0x1234, actual=0x5678
```
**含义**: 数据传输错误  
**检查**: 串口线质量、电磁干扰、波特率设置

#### 配置验证失败
```
[2026-04-23 10:00:25.789] WARN - Station configuration validation failed:
[2026-04-23 10:00:25.790] WARN -   - aqmd.baudRate: Invalid baud rate
```
**含义**: 配置参数不合法  
**检查**: 配置文件参数值

## 8. 应急处理

### 8.1 紧急停止

**操作步骤**:
1. 点击界面上的"停止"按钮
2. 如无响应，关闭应用程序窗口
3. 如仍无响应，使用任务管理器结束进程
4. 断开设备电源

### 8.2 数据恢复

**配置文件损坏**:
```batch
# 恢复备份
copy config\station.json.backup config\station.json
```

**报告文件丢失**:
- 检查`reports/`目录
- 查看回收站
- 从备份恢复

### 8.3 系统重置

**完全重置步骤**:
1. 备份当前配置和数据
2. 卸载应用程序
3. 删除安装目录
4. 重新安装
5. 恢复配置文件
