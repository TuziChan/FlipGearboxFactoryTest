## 1. 概述

### 1.1 目标

本文档定义 FlipGearboxFactoryTest 上位机项目从需求输入到生产部署的**完整开发工作流**。它不是一份抽象的流程图，而是可执行的纪律：每个阶段都有明确的输入、输出、准入条件、退出标准和具体命令。

### 1.2 适用范围

- **适用人员**：所有参与本项目的 AI Agent 和人类开发者
- **适用场景**：新功能开发、缺陷修复、架构重构、设备适配、UI 迭代
- **不适用场景**：紧急热修复（见 [9.3 热修复流程](#93-热修复流程)）

### 1.3 项目上下文

本项目是一个基于 Qt6/QML 的工业上位机，管理多条 RS485/Modbus RTU 总线上的设备，执行齿轮箱出厂测试流程。核心特征：

- **双运行模式**：仿真模式（无硬件）/ 真实模式（连接物理设备）
- **实时性要求**：33ms 控制周期（30Hz）
- **多总线拓扑**：一台 PC 管理多条 RS485 总线，每条总线一主多从
- **MVVM 架构**：QML 纯展示，C++ 承载 ViewModel、Domain 和 Infrastructure
- **团队配置**：SpectrAI 多 Agent 协作（Leader + 领域/VM/UI/基础设施/协议/测试/构建工程师）

## 2. 开发铁律与原则

### 2.1 四条绝对铁律

| 铁律 | 含义 | 违反后果 |
|------|------|----------|
| **NO FIX WITHOUT ROOT CAUSE** | 遇到错误必须执行系统化调试查明根因，严禁直接给代码修复 | 引入表面修复，根因复发 |
| **NO PRODUCTION CODE WITHOUT RED TEST** | 严禁在测试失败前写生产代码 | 代码不可测试、不可验证 |
| **NO BLIND MOCKING** | 严禁测试 Mock 行为，必须测试真实行为 | 测试通过但真实场景失败 |
| **NO GUESSING THE OUTPUT** | 严禁未实际运行命令并看到成功输出就宣布完成 | 虚假完成状态 |

### 2.2 架构纪律

1. **单向依赖**：UI → ViewModel → Domain → Infrastructure，禁止反向依赖
2. **设备抽象**：Domain 层只依赖设备能力接口（`IMotorDriveDevice` 等），不依赖具体总线
3. **配置驱动**：改 `config/station.json` 即可换串口/站号/工位，不依赖重新编译
4. **QML 禁区**：QML 禁止直接访问串口、禁止直接做 Modbus 读写、禁止直接计算测试判定
5. **并发安全**：共享数据必须加锁（`QMutex` + `QMutexLocker`），禁止长时间持锁

### 3.2 工作流程

```
Step 1: 需求澄清会议（Leader 主持，全角色参与）

  ├── 明确非功能性需求（性能、分辨率、兼容性）

  ├── 确认修改范围属于哪个分层
  ├── 确认是否影响设备接口（如需新增接口方法）
  ├── 确认配置格式变更
  ├── 绘制时序图（涉及多设备协调时）

  ├── 全角色参与评审
  ├── 重点检查：是否破坏单向依赖、是否新增硬件耦合
```

### 3.3 输出清单

|--------|----------|----------|
| 架构设计文档 | `Docs/superpowers/specs/YYYY-MM-DD-<feature>-design.md` | 目标、范围、分层修改点、接口变更、验收标准 |

### 3.4 准入/退出条件

- **准入条件**：需求输入明确，相关设备手册已就位

## 4. 阶段二：任务拆分与规划

### 4.1 拆分原则

采用 **Milestone + Task** 两级拆分：

- **Milestone**：按架构分层或功能模块划分，有明确的交付物和验收标准
- **Task**：每个 Milestone 拆分为可独立执行、不超过 4 小时的任务

### 4.2 标准 Milestone 模板

```markdown
## Milestone X: <名称>

**目标**: <一句话描述>
**负责人**: <角色ID>
**预计工期**: <天数>
**交付物**:
- [ ] 文件1
- [ ] 文件2
- [ ] 测试通过

**验收标准**:
1. <可验证的条件1>
2. <可验证的条件2>

**依赖**: <前置 Milestone>
```

### 4.3 任务分配流程

```
Leader 创建 Milestone → 拆分为 Task → 分配至角色 → 角色认领 → 执行 → 完成报告
```

**分配规则**：
- 一个 Task 只分配给一个角色
- 跨层修改必须拆分为多个 Task（如 Domain 变更 + ViewModel 适配 + UI 绑定）
- 涉及硬件的 Task 必须标注 `[HW_REQUIRED]` 或 `[MOCK_FIRST]`

### 4.4 角色职责映射

| 角色ID | 职责范围 | 典型产出 |
|--------|----------|----------|
| `leader` | 任务分配、进度协调、风险升级 | Milestone 计划、任务分配 |
| `domain-dev` | `src/domain/` 业务逻辑、状态机、判定规则 | `GearboxTestEngine`、RecipeValidator |
| `viewmodel-dev` | `src/viewmodels/` Q_PROPERTY、命令、数据转换 | `TestExecutionViewModel` |
| `ui-dev` | `src/ui/` QML 页面、组件、布局 | `TestExecutionPage.qml`、组件 |
| `infra-dev` | `src/infrastructure/services/`、`src/infrastructure/config/` | `HistoryService`、`ConfigLoader` |
| `protocol-dev` | `src/infrastructure/bus/`、`src/infrastructure/devices/` | `ModbusRtuBusController`、设备适配器 |
| `tester` | `tests/` 测试用例、Mock 对象、覆盖率 | `*Tests.cpp`、`Mock*.h` |
| `builder` | `CMakeLists.txt`、构建脚本、打包 | 构建配置、CI 脚本 |
| `reviewer` | 代码审查、规范检查、质量门禁 | 审查意见、LGTM/拒绝 |

## 5. 阶段三：迭代开发工作流（核心）

### 5.1 开发周期模型

采用 **测试驱动开发（TDD）+ 短迭代** 模型：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   红阶段     │ → │   绿阶段     │ → │  重构阶段    │ → │   提交阶段   │
│  写失败测试  │    │ 写最少代码   │    │  清理代码    │    │  本地验证    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     ↑___________________________________________________________|
                         （下一个测试用例）
```

### 5.2 红阶段：编写失败测试

**准入条件**：
- 设计文档已明确本 Task 的接口和行为
- 相关 Mock 对象已存在（如测试设备接口）

**执行步骤**：

1. **确定测试文件位置**：
   - Domain 测试 → `tests/domain/`
   - ViewModel 测试 → `tests/viewmodels/`
   - 基础设施测试 → `tests/infrastructure/`
   - 设备测试 → `tests/devices/`
   - 仿真测试 → `tests/simulation/`
   - UI 测试 → `tests/ui/`

2. **编写第一个失败测试**：

```cpp
// tests/domain/test_GearboxTestEngine.cpp
#include <QTest>
#include "src/domain/GearboxTestEngine.h"
#include "tests/mocks/MockDevices.h"

class GearboxTestEngineTests : public QObject {
    Q_OBJECT
private slots:
    void test_startTest_invalidSn_blocksStart() {
        // Arrange
        Domain::GearboxTestEngine engine;
        MockMotorDriveDevice motor;
        MockTorqueSensorDevice torque;
        MockEncoderDevice encoder;
        MockBrakeDevice brake;
        engine.setDevices(&motor, &torque, &encoder, &brake);
        
        // Act
        bool result = engine.startTest("");  // 空 SN
        
        // Assert
        QVERIFY(!result);
        QCOMPARE(engine.currentState().phase, Domain::TestPhase::Idle);
    }
};
QTEST_MAIN(GearboxTestEngineTests)
#include "test_GearboxTestEngine.moc"
```

3. **运行测试确认失败**：

```powershell
# 配置
$env:PATH = 'F:\Qt\6.11.0\mingw_64\bin;F:\Qt\Tools\mingw1310_64\bin;' + $env:PATH
cmake -S . -B build\mingw-debug -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Debug -DCMAKE_PREFIX_PATH=F:/Qt/6.11.0/mingw_64

# 构建并运行单个测试
cmake --build build\mingw-debug --parallel --target DomainEngineTests
ctest --test-dir build\mingw-debug -R DomainEngineTests --output-on-failure
```

4. **记录失败输出**：截图或复制错误信息，作为下一步的驱动

### 5.3 绿阶段：编写最少生产代码

**纪律**：
- 只允许写让当前测试通过的最少代码
- 禁止同时实现多个测试期望的功能
- 禁止优化、禁止提前抽象

**示例**：

```cpp
// src/domain/GearboxTestEngine.cpp
bool GearboxTestEngine::startTest(const QString& serialNumber) {
    if (serialNumber.isEmpty()) {
        return false;  // 最少代码，仅让测试通过
    }
    // TODO: 后续测试驱动实现
    return true;
}
```

### 5.4 重构阶段

**触发条件**：当前测试通过，但代码有坏味道

**允许的重构**（不改变行为）：
- 提取函数/变量
- 重命名（使用 IDE 重构工具）
- 消除重复
- 改善命名

**禁止的重构**（会改变行为）：
- 新增功能
- 修改接口签名
- 修改算法逻辑

**每次重构后必须**：
```powershell
ctest --test-dir build\mingw-debug -R <当前测试套件> --output-on-failure
```

### 5.5 提交阶段

**提交前检查清单**：

- [ ] 所有新增/修改的测试通过
- [ ] `qmllint` 无新警告（UI 修改时）
- [ ] 编译无警告（`-Wall -Wextra` 级别）
- [ ] 代码符合编码规范
- [ ] 无临时调试代码（`qDebug` 已清理或转为 `qInfo`/`qWarning`）

**提交命令**：

```powershell
# 1. 确认改动范围
git status
git diff

# 2. 提交（遵循 commit message 规范）
git add <相关文件>
git commit -m "feat(domain): 添加 GearboxTestEngine startTest SN 校验

- 空 SN 阻止测试启动
- 添加 GearboxTestEngineTests 首组测试
- 通过 DomainEngineTests

Refs: task-xxx"
```

**Commit Message 规范**：

```
<type>(<scope>): <subject>

<body>

Refs: <task-id>
```

| Type | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | 缺陷修复 |
| `test` | 测试相关 |
| `refactor` | 重构 |
| `docs` | 文档 |
| `chore` | 构建/工具链 |
| `perf` | 性能优化 |

### 5.6 跨层开发顺序

当一个功能需要修改多个分层时，**必须从底层向上开发**：

```
Infrastructure（设备接口/配置加载）
     ↓
Domain（业务逻辑/状态机）
     ↓
ViewModel（数据转换/命令暴露）
     ↓
UI（QML 绑定/交互）
```

**原因**：上层依赖下层接口。先定下下层接口，上层才能基于真实接口开发，避免 Mock 接口与实际实现脱节。

### 5.7 仿真优先原则

**所有涉及硬件的功能必须先在仿真模式下实现和验证**：

1. 实现 `SimulatedXxxDevice` 对应真实设备的行为
2. 在仿真模式下通过完整测试流程
3. 记录仿真与真实设备的预期差异
4. 在真实硬件上验证并修正

**仿真模式启动**：

```powershell
.\build\mingw-debug\bin\appFlipGearboxFactoryTest.exe --mock
```

## 6. 阶段四：代码审查与质量门禁

### 6.1 审查触发条件

以下情况**必须**经过代码审查：

- 任何修改 `src/domain/` 的提交
- 任何修改 `src/infrastructure/devices/` 的提交
- 任何修改 `src/infrastructure/bus/` 的提交
- 任何新增/修改公共接口的提交
- 任何删除测试的提交
- Milestone 完成前的最后一次提交

### 6.2 审查清单

**架构层审查**：
- [ ] 是否破坏单向依赖？
- [ ] Domain 层是否引入了基础设施细节？
- [ ] 设备接口是否按能力抽象而非页面控件抽象？

**正确性审查**：
- [ ] 边界条件是否处理？（空值、越界、超时）
- [ ] 并发安全是否正确？（锁的粒度和范围）
- [ ] 错误处理是否完整？（不静默吞异常）

**可测试性审查**：
- [ ] 是否有对应的失败测试先于生产代码？
- [ ] 测试是否验证了真实行为而非 Mock 行为？
- [ ] 测试是否独立、不依赖执行顺序？

**代码风格审查**：
- [ ] 命名是否符合规范？
- [ ] 是否有魔法数字？（应提取为命名常量）
- [ ] QML 是否有未限定访问警告？

### 6.3 审查流程

```
开发者提交 PR → Reviewer 48h 内响应 → 修改意见 → 开发者修正 → Reviewer 确认 → 合并
```

**审查意见分级**：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **Blocker** | 必须修改，否则禁止合并 | 修正后重新审查 |
| **Major** | 建议修改，影响可维护性 | 修正或提供合理解释 |
| **Minor** | 风格/建议 | 可选处理 |
| **Nit** | 挑剔性意见 | 可选处理 |

### 6.4 自动化门禁

**必须在合并前通过以下检查**：

```powershell

cmake --build build\mingw-debug --parallel

# 2. 全量测试通过
ctest --test-dir build\mingw-debug --output-on-failure

F:\Qt\6.11.0\mingw_64\bin\qmllint.exe src\ui\pages\<修改的文件>.qml
```

## 7. 阶段五：硬件集成验证

### 7.1 硬件验证前提

- 仿真模式下该功能已通过全部测试
- 设备手册寄存器映射已核对
- `config/station.json` 已配置正确的 COM 口和站号

### 7.2 硬件验证环境

**推荐硬件验证拓扑**：

```
PC (上位机)
  ├── COM3 → RS485 → AQMD3610NS-A2 (电机驱动, 站号1)
  ├── COM4 → RS485 → DYN-200 (扭矩传感器, 站号2)
  ├── COM5 → RS485 → 单圈编码器 (站号3)
  └── COM6 → RS485 → 双通道数控电源 (站号4, 制动用)
```

### 7.3 硬件验证流程

**Step 1: 单设备通信验证**

```powershell
# 启动真实模式
.\build\mingw-debug\bin\appFlipGearboxFactoryTest.exe

# 验证设备在线状态
# 观察底部状态栏：所有设备应显示绿色在线点
```

**Step 2: 寄存器读写验证**

使用 Modbus 调试工具（如 Modbus Poll）或项目内置诊断页面：
- 读取 Holding Register 确认返回预期值
- 写入单个寄存器，读取确认写入成功
- 测试超时场景（拔掉设备线，确认报错而非挂死）

**Step 3: 单阶段验证**

| 阶段 | 验证方法 | 预期结果 |
|------|----------|----------|
| Homing | 观察电机找零 + 磁铁检测 | 电机停止在零位，AI1 触发 |
| Idle | 观察正反转空载运行 | 速度稳定，扭矩接近零 |
| Angle | 观察角度定位 | 5 个定位点角度在容差内 |
| Load | 观察制动电流 ramp | 电流递增，锁止检测触发 |
| Return | 观察回零 | 最终停在零位 |

**Step 4: 异常场景验证**

- 断开一个设备 → UI 显示通信失败，不判为产品 NG
- 错误站号 → 明确提示"站号 X 无响应"
- 中途急停 → 电机立即停止，制动电源断开

## 10. 阶段八：版本管理与发布

### 10.1 版本号规范

采用 **SemVer**（语义化版本）：`MAJOR.MINOR.PATCH`

| 位 | 递增条件 |
|----|----------|
| MAJOR | 不兼容的 API 变更、协议变更 |
| MINOR | 新增功能（向下兼容） |
| PATCH | 缺陷修复、性能优化 |

### 10.2 分支模型

```
main (保护分支，永远可发布)
  ├── release/v1.0.x (发布分支，只接受 bugfix)
  ├── develop (集成分支，日常开发合并)
  │     ├── feature/xxx (功能分支)
  │     ├── fix/xxx (修复分支)
  │     └── refactor/xxx (重构分支)
```

### 10.3 发布流程

```
1. 从 main 切出 release/vX.Y.Z
2. 运行全量测试 + 硬件验收
3. 修复发现的缺陷（cherry-pick 到 develop）
4. 更新版本号（CMakeLists.txt、QML 标题）
5. 编写 Release Notes
6. 合并 release 到 main，打标签 vX.Y.Z
7. 执行 windeployqt 打包
8. 上传发布包 + Release Notes
```
